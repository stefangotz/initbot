# Web App Login and Session Behaviour

## Login flows

### Standalone join

Visit `/<prefix>/join/`, enter a display name, and click **Join**.

The display name is looked up case-insensitively in the player table.

- If no player with that name exists, a new player record is created.
- If a standalone player with that name already exists, that existing record is reused.
- If a Discord-linked player with that name exists, the join is rejected with an error
  — the name is reserved for the Discord identity.

On success the browser receives a signed session cookie and is redirected to the tracker.

### Discord `$web` command

1. A player types `$web` in any channel the bot is in, or in a DM to the bot.
2. The bot generates a random, single-use token, writes it to the shared database with a
   60-second expiry, constructs a login URL, and sends it to the player via Discord DM.
3. The player opens the link. A GET request to `/<prefix>/<token>/` validates the token
   without consuming it, then serves a login page that immediately auto-submits a POST.
   (The GET step exists because Discord's link-preview crawler issues GET requests; consuming
   the token on GET would break the login for the actual user.)
4. The POST consumes the token and creates a signed session cookie.

## Player identity

### Standalone players

The display name entered at `/join/` is the player's identity. Two visits with the same
name (case-insensitive) produce the same player record and therefore the same `player_id`
in the session. This means:

- **Returning after closing the browser** works: the player gets the same record back and
  their characters (stored against their `player_id`) are still associated with them.
- **Joining on a second device or browser** works: both sessions carry the same `player_id`
  and are effectively two views of the same player identity.
- **Two browser windows open simultaneously** works: each window has its own session cookie
  and its own sort state, but both show the same player name and both see all the same
  tracker updates via SSE.

There is no password or secret behind the name. Access control rests entirely on knowing the
URL prefix. Within that boundary all users are trusted, so name-based identity is sufficient.

### Discord players

A Discord player's identity is their Discord snowflake ID, which the bot obtains from the
`$web` command context. The display name shown in the tracker is their Discord display name
at the time they last used `$web`, and is updated on each login.

## Access model

All web app endpoints sit under a hard-to-guess URL prefix (`web_url_path_prefix`). Without
knowing the prefix there is nothing to reach, so the application is invisible to port scanners
and opportunistic probes.

For the standalone join flow, the prefix is the entire access control boundary. Anyone who
knows it can join under any display name. No account system, password storage, or OAuth
integration is needed — within the boundary all users are trusted.

For the Discord `$web` flow, the prefix is still required to reach the login endpoint, and
the one-time token provides an additional layer: Discord provides the sender's identity
cryptographically, the DM channel is private to the player and the bot, and the token expires
in 60 seconds and can only be used once.

## Session lifetime

Sessions have a fixed time-to-live of **eight hours** from the moment of login. This applies
equally to standalone join sessions and Discord `$web` sessions.

- The expiry timestamp is written into the session cookie at login and is not renewed by
  subsequent requests. There is no sliding expiry and no activity-based renewal.
- Once the session expires every protected endpoint returns 403. The player must log in again
  — either by visiting `/join/` or by using `$web` — to get a fresh session.
- There is no server-side session store. The session is entirely contained in the signed
  cookie. The server has no record of which sessions are open or how many there are.

## Session signing key

Sessions are signed cookies. The signing key is generated by the web app and stored in the
shared SQLite database. It also has an eight-hour TTL.

- If the web app is restarted within the key's validity window, the existing key is reused
  and all previously issued sessions remain valid.
- If the key has expired (or the database is cleared), a new key is generated. All previously
  issued session cookies become unverifiable and are silently treated as absent, which causes
  a 403 on the next request. Players must log in again.

In practice the two eight-hour windows are aligned: a session cookie issued just before a key
rotation will expire from key rotation at roughly the same time it would have expired from its
own TTL anyway.
